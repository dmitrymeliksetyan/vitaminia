# DEPLOY.md — инфраструктура MEDIZIN.RU (Infrastructure v2)

Этот документ описывает архитектуру публикации после миграции с Cloudflare
Pages на собственный сервер (Node + nginx + PM2 + GitHub Actions). Пишется
для будущего себя/другого администратора — предполагается, что читающий
не участвовал в миграции и не помнит контекст.

## 1. Архитектура

Один Astro-проект (`output: "hybrid"`, адаптер `@astrojs/node`, режим
`standalone`), одна команда `npm run build` — но результат физически
разделён на две независимые части:

- **Статика** — `dist/client/` (126+ HTML-страниц: симптомы, лекарства,
  статьи, sitemap, robots.txt — весь SEO-контент). Отдаётся nginx
  напрямую с диска, Node вообще не участвует в этих запросах.
- **SSR** — `dist/server/entry.mjs` (самодостаточный Node HTTP-сервер).
  Обслуживает ТОЛЬКО `/assistant`, `/my`, `/admin`, `/api` и `/health` —
  всё, что требует Supabase-сессии, авторизации или живых AI-вызовов.

Почему не два отдельных проекта/репозитория: при ~150+ существующих
страницах полноценный monorepo-сплит на этом этапе был бы рискованным
переписыванием ради архитектурной чистоты без практической пользы — тот
же результат ("публикация статики не трогает Node и наоборот") достигается
на уровне деплой-скриптов через сравнение SHA-256 хэша `dist/server` между
релизами (см. §3 и `scripts/deploy/deploy.sh`): если SSR-бандл не менялся,
PM2 не перезапускается, только переключается symlink статики.

```
Пользователь
   │
   ▼
 nginx (порт 80/443)
   ├── /health, /assistant, /my, /admin, /api  → 127.0.0.1:4321 (Node, PM2)
   └── всё остальное                            → статика с диска
                                                    (current/dist/client)
```

### 1.1. AI Worker (medizin-worker) — ОТДЕЛЬНЫЙ проект, не часть этого деплоя

Этап "Выделение AI Worker в отдельный независимый сервис": весь AI-конвейер
производства контента (research → draft → medical_review → final_review →
seo_review), AI-стратег и реальная публикация в GitHub (git commit) БОЛЬШЕ
НЕ являются частью `medizin` (этого репозитория) и НЕ описываются этим
документом. Они живут в отдельном проекте `medizin-worker` — со своим
собственным репозиторием, своим `package.json`, своим `.env`
(`SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `ANTHROPIC_API_KEY`,
`GITHUB_TOKEN`, `GITHUB_REPO`, `GITHUB_BRANCH`), своим PM2-конфигом и своей
инструкцией по запуску/обновлению/откату (см. `medizin-worker/README.md`).

Worker может физически находиться где угодно — на этом же сервере, на
отдельном VPS, в США/Канаде/Европе, на домашнем сервере — это НЕ имеет
значения, потому что единственный канал связи между `medizin` (SSR) и
`medizin-worker` — таблицы Supabase (`content_jobs`, `content_strategy_runs`
и связанные). SSR никогда не делает HTTP-запросов к Worker и никогда не
узнаёт, где он запущен и запущен ли вообще:

```
Пользователь → nginx → medizin-ssr → Supabase ← medizin-worker
                                                   (где угодно физически)
```

Если Worker выключен — сайт (включая `/admin/editorial`) продолжает
работать полностью нормально: показывает уже опубликованные материалы и
текущее состояние очереди ("ждёт обработки"/"в работе"/"готово"), просто
ничего нового не производится и не публикуется, пока Worker не запустится
снова. `medizin-ssr` (этот проект) в принципе не содержит кода, который бы
исполнял AI-вызовы или писал в GitHub — соответствующие файлы
(`src/lib/content-editor/{run-stage,pipeline,ai-client,github-client,...}.ts`,
`src/lib/content-registry/{validate,search,links,seo-health}.mjs` и т.д.)
удалены из этого репозитория (см. финальный отчёт этапа — список удалённых
файлов). Общий код (валидация Content Registry, поиск дублей, нормализация
job-полей и т.п.), нужный ОБОИМ проектам, вынесен в третий пакет
`medizin-shared`, на который `medizin` и `medizin-worker` ссылаются как на
локальную npm-зависимость (`file:../medizin-shared`) — не копия кода.

Развёртывание/обновление/откат Worker'а, запуск, переменные окружения —
см. `medizin-worker/README.md` в его собственном репозитории. Этот документ
(DEPLOY.md) описывает ТОЛЬКО деплой сайта (medizin-ssr + статика).

## 2. Структура каталогов на сервере

```
/var/www/medizin/
├── releases/
│   ├── 20260710120000/     ← полный checkout + build одного релиза
│   ├── 20260712083000/
│   └── 20260713093000/
├── current -> releases/20260713093000/   ← symlink, атомарно переключается
├── shared/
│   ├── .env.production      ← секреты, переживают все релизы
│   └── logs/                ← логи PM2 (pm2-out.log, pm2-error.log)
└── ecosystem.config.cjs     ← конфиг PM2 (см. §6, ставится один раз)
```

Именно `current` — то, на что смотрит и nginx (`root
/var/www/medizin/current/dist/client`), и PM2 (`cwd:
${APP_DIR}/current` в `ecosystem.config.cjs`).

## 3. Публикация — что происходит автоматически

Триггер — **любой push в ветку `main`**, включая автоматические коммиты
AI-редакции (`src/lib/content-editor/github-client.ts` коммитит `.mdx` +
обновление Registry через GitHub Git Data API — этот шаг не изменился,
изменилось только то, что происходит ПОСЛЕ push).

1. **GitHub Actions** (`.github/workflows/deploy.yml`) подхватывает push,
   собирает проект на раннере (`scripts/deploy/build.sh`: `npm ci` +
   `npm run build`, генерирует `build-info.ts` с реальным commit SHA).
2. Готовый `dist/` + `package.json`/`package-lock.json` + `scripts/`
   переносятся по `rsync` через SSH в новый
   `releases/<timestamp>/` на сервере.
3. CI по SSH запускает `scripts/deploy/deploy.sh <timestamp>` **на
   сервере**. Этот скрипт:
   - ставит прод-зависимости (`npm ci --omit=dev`) внутри нового релиза;
   - **пробно запускает новый релиз на отдельном порту** (4322) и
     прогоняет `verify.sh` против него — если что-то не так, `current`
     вообще не трогается, сайт продолжает работать на старой версии
     (zero downtime, атомарность — п.3-4 ТЗ);
   - если проверка прошла — атомарно переключает `current` (`ln -sfn`,
     атомарная операция на POSIX — не бывает промежуточного состояния
     "current не существует");
   - сравнивает SHA-256 `dist/server` старого и нового релиза — если
     SSR-бандл не менялся (публикация была только контентная), PM2 **не**
     перезапускается вообще;
   - если бандл менялся — `pm2 reload --update-env` (мягкий reload, не
     roll restart с простоем);
   - прогоняет `verify.sh` ещё раз, уже через настоящий `nginx`/публичный
     URL — если и это не прошло, автоматически откатывается
     (`rollback.sh`);
   - **сразу после успешной публичной проверки запускает
     `scripts/deploy/resolve-deploying-jobs.mjs`** — переводит все
     материалы AI-редакции из статуса "deploying" в "published" (проверяет
     реальный публичный URL каждого такого материала). Раньше (во времена
     Cloudflare Pages) этот переход происходил ТОЛЬКО если админ держал
     открытым экран job'а в браузере (клиентский поллинг) — теперь деплой
     сам знает точный момент публикации и закрывает статус без какого-либо
     браузера. Best-effort и некритично для деплоя: если этот шаг
     не сработал, сайт всё равно уже опубликован, а клиентский поллинг в
     админке остаётся рабочим резервным механизмом;
   - удаляет релизы старше `KEEP_RELEASES` (по умолчанию 5).

Никакого ручного `git pull`/`npm ci`/`pm2 restart` на сервере — это то,
что явно требовал п.13 ТЗ ("Автоматический деплой"). Полный путь статьи:
`AI → git commit → GitHub Actions → build → deploy.sh → current → PM2
reload → resolve-deploying-jobs.mjs → статья "Опубликовано" в админке` —
без единого ручного действия на любом шаге.

**Важная деталь, проверенная живым тестом:** `dist/server/entry.mjs`
включает в себя манифест ВСЕХ маршрутов сайта (генерируется Astro), поэтому
SHA-256 нового и старого `dist/server` практически ВСЕГДА отличаются даже
при чисто контентной публикации (одна новая статья уже меняет манифест) —
на практике `pm2 reload` срабатывает почти на каждый деплой, а не только
при реальных изменениях кода `/assistant /my /admin /api`. Это безопасно
(лишний reload не хуже пропущенного), просто ожидаемое поведение, а не
редкий edge-case.

**Ещё одна деталь для отладки:** Node-адаптер (`@astrojs/node`) в
`standalone`-режиме умеет сам отдавать статику из `dist/client` (это то,
что позволяет `npm run preview` работать без nginx) — но живым тестом
подтверждено, что уже запущенный процесс продолжает отдавать СТАРУЮ
статику после переключения symlink `current`, пока сам процесс не
перезапущен (`pm2 reload`). В production это не проблема — статику
всегда отдаёт nginx напрямую с диска (см. `nginx/medizin.conf`), а не
Node, и nginx открывает файлы заново на каждый запрос. Но если
когда-нибудь тестируете публикацию локально БЕЗ nginx (`npm run preview`
+ ручное переключение `current`) — не удивляйтесь, что новая статья не
появится, пока не перезапущен сам процесс.

## 4. Rollback

Вручную:
```bash
npm run deploy:rollback
# или на сервере напрямую:
bash scripts/deploy/rollback.sh              # откат на предыдущий релиз
bash scripts/deploy/rollback.sh 20260710120000  # откат на конкретный релиз
```

Автоматически: `deploy.sh` сам вызывает `rollback.sh`, если финальная
проверка после переключения `current` не прошла (см. §3, последний шаг).

`rollback.sh` определяет "предыдущий" релиз по времени модификации
каталога внутри `releases/` (если не передан явный timestamp), атомарно
переключает symlink обратно, перезапускает PM2, проверяет результат.

## 5. Health Check

`GET /health` (реализация — `src/pages/health.ts`, Node SSR, не
кэшируется) отдаёт JSON:

```json
{
  "status": "ok" | "degraded",
  "node": { "ok": true, "version": "v20.x" },
  "database": { "ok": true, "latencyMs": 42 },
  "git": { "sha": "abc1234", "branch": "main" },
  "build": { "version": "20260713.0930", "buildTime": "..." },
  "environment": "production",
  "uptimeSeconds": 12345
}
```

`database` — лёгкий ping в PostgREST (`{SUPABASE_URL}/rest/v1/` с
anon-ключом, таймаут 2с) — проверяет "база физически отвечает", не
конкретную бизнес-таблицу (устойчиво к будущим изменениям схемы).

Локально: `npm run health:check` (ожидает уже запущенный сервер — `npm
run preview` или `node scripts/deploy/run-server.mjs`).

`/health` — единственное исключение из правила "Node отдаёт только
/assistant /my /admin /api": он не является SEO-страницей и физически не
может быть статикой (Uptime и статус БД вычисляются в момент запроса).
В nginx-конфиге он проксируется отдельным точечным `location = /health`
(см. `nginx/medizin.conf`).

## 6. Первичная настройка сервера (один раз, вручную)

Всё, что описано в §3, автоматизирует ПОВТОРНЫЕ публикации. Перед самой
первой публикацией на новом сервере нужно один раз вручную:

1. Установить Node.js ≥20, nginx, PM2 (`npm install -g pm2`).
2. Создать структуру каталогов:
   ```bash
   sudo mkdir -p /var/www/medizin/{releases,shared/logs}
   sudo chown -R $USER:$USER /var/www/medizin
   ```
3. Заполнить `/var/www/medizin/shared/.env.production` (скопировать из
   `.env.example`, вставить реальные секреты сайта — `ANTHROPIC_API_KEY`
   Помощника, `SUPABASE_SERVICE_ROLE_KEY` и т.д. — GitHub-секретов здесь
   больше нет, см. §1.1: они у `medizin-worker`, в его собственном `.env`).
4. Скопировать `ecosystem.config.cjs` из репозитория в
   `/var/www/medizin/ecosystem.config.cjs` (не пересобирается при каждом
   деплое — это стабильный, "серверный" конфиг, а не часть релиза).
5. Установить и прописать `nginx/medizin.conf` в
   `/etc/nginx/sites-available/`, симлинк в `sites-enabled/`, `nginx -t &&
   systemctl reload nginx`. Выпустить TLS-сертификат (certbot).
6. Сгенерировать SSH-ключ для деплоя, публичный добавить в
   `~/.ssh/authorized_keys` на сервере, приватный — в секрет
   `DEPLOY_SSH_KEY` репозитория на GitHub (Settings → Secrets and
   variables → Actions). Также завести секреты `DEPLOY_HOST`,
   `DEPLOY_USER`, `DEPLOY_APP_DIR`, `PUBLIC_SUPABASE_URL`,
   `PUBLIC_SUPABASE_ANON_KEY`.
7. Первый деплой можно прогнать вручную, чтобы создать самый первый
   релиз (дальше CI справится сам):
   ```bash
   bash scripts/deploy/build.sh
   rsync -az dist package.json package-lock.json scripts \
     user@server:/var/www/medizin/releases/$(date -u +%Y%m%d%H%M%S)/
   ssh user@server "bash /var/www/medizin/releases/<timestamp>/scripts/deploy/deploy.sh <timestamp>"
   ```

## 7. Восстановление после ошибки

- **Сайт отдаёт 502/503 после публикации** — `deploy.sh` должен был
  откатиться сам (см. §3), но если нет: `bash
  scripts/deploy/rollback.sh` вручную, затем разбираться в
  `shared/logs/pm2-error.log` и в выводе `deploy.sh` (он всё логирует
  через `log()` из `config.sh`, формат `[ISO-время] сообщение`).
- **`npm ci --omit=dev` падает в новом релизе** — обычно рассинхрон
  `package-lock.json`; `current` не тронут (см. §3, деплой останавливается
  до переключения symlink) — сайт продолжает работать. Чинить
  `package-lock.json` и пушить заново.
- **`verify.sh` падает на конкретной проверке** — вывод скрипта прямо
  называет, какая проверка не прошла (`FAIL <label> (<url> -> ожидался X,
  получили Y)`) — не нужно гадать.
- **PM2-процесс не поднимается** — `pm2 logs medizin-ssr`, проверить, что
  `shared/.env.production` реально существует и просимлинкован в текущий
  релиз (`deploy.sh` делает это сам, но если `shared/.env.production`
  отсутствует физически — будет явное предупреждение в логе деплоя).
- **Материалы не двигаются по этапам, хотя админка показывает "В работе"**
  — это НЕ проблема этого репозитория/деплоя (см. §1.1: `medizin-worker` —
  отдельный проект на отдельном деплое, возможно на другом сервере). Нужно
  проверять состояние ИМЕННО Worker'а на ЕГО сервере (`pm2 status`,
  `pm2 logs medizin-worker` — см. `medizin-worker/README.md`), а не
  что-либо в этом репозитории. Сайт (medizin-ssr) сам по себе никогда не
  является причиной "материалы не двигаются" — он только читает статус.
- **`medizin-ssr` не видит `SUPABASE_SERVICE_ROLE_KEY` и подобные секреты,
  пока их не выполнить вручную через `source .env.production` перед
  стартом PM2** — это симптом того, что процесс запущен в обход
  `deploy.sh` (тот сам создаёт симлинк `<релиз>/.env.production ->
  shared/.env.production`, см. шаг "Секреты релиза"). `scripts/deploy/
  run-server.mjs` через `scripts/deploy/load-env.mjs` ищет секреты И по
  релиз-специфичному пути, И по стабильному `APP_DIR/shared/.env.production`
  — ручной `source` не должен требоваться. Если проблема повторяется —
  проверить, что `/var/www/medizin/shared/.env.production` реально
  существует и читаем пользователем, от имени которого работает PM2 (`ls
  -la /var/www/medizin/shared/.env.production`), и посмотреть в `pm2
  logs`, какой именно путь (или что ни один) вывел `[run-server] Загружен
  ...`.
- **CI (GitHub Actions) не может достучаться по SSH** — проверить секреты
  `DEPLOY_HOST`/`DEPLOY_USER`/`DEPLOY_SSH_KEY`, а также что публичный ключ
  реально в `authorized_keys` на сервере.

## 8. Версия сайта

Каждая сборка генерирует `src/generated/build-info.ts`
(`scripts/generate-build-info.mjs`, запускается автоматически перед `dev`
и `build`) со схемой версии `YYYYMMDD.HHmm` (UTC, монотонно растёт,
согласовано как альтернатива semver — в эфемерном CI-раннере негде хранить
персистентный счётчик). Отображается компонентом
`src/components/admin/BuildInfoBadge.tsx` в `/admin` (компактный бейдж +
разворачиваемая модалка: версия/коммит/ветка/окружение/время
сборки/версии Astro и Node) и присутствует в ответе `/health`.

## 9. Что сознательно НЕ входит в этот этап

Дизайн, главная страница, каталог симптомов/лекарств, Помощник, «Моя
карта», функциональность AI-редакции — не менялись (см. п.16 исходного
ТЗ). Это инфраструктурный этап: скорость, надёжность, автоматизация
публикации.
