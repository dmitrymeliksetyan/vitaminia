# MEDIZIN — Стандарт страницы симптома

Этот документ описывает, как создавать контент для проекта.
Одна страница = один файл `.mdx` в `src/content/symptoms/{категория}/{slug}.mdx`.

---

## Структура frontmatter

```yaml
title: Высокая температура           # H1 страницы и заголовок в поиске
slug: high-fever                      # должен совпадать с именем файла
category: temperature                 # должен совпадать с папкой и slug в categories/

shortAnswer: >                        # 1-2 предложения — первый экран, 30 секунд
  Температура выше 38.5°C — реакция
  организма на инфекцию.

keyPoints:                            # «Главное» — 3-5 пунктов
  - 38.5°C и выше считается высокой
  - Чаще всего это вирус

causes:                               # Возможные причины — список
  - ОРВИ и грипп
  - Бактериальные инфекции

selfCare:                             # Что можно сделать дома
  - Пить больше жидкости
  - Отдыхать

whenToSeeDoctor:                      # Несрочные показания к врачу
  - Температура держится более 3 дней

whenUrgent:                           # Красные флаги — срочная помощь
  - Температура выше 40°C
  - Спутанность сознания

severity: high                        # low | medium | high

tags:
  - fever
  - infection

manualRelated:                        # опционально — ручные ссылки (slug'и)
  - child-fever

updated: "2026-06-01"                 # дата последнего обновления
reviewed: "2026-06-01"               # опционально — дата медицинской проверки

sources:                              # опционально
  - title: ВОЗ — лихорадка
    url: https://www.who.int/...

seoTitle: >                           # опционально — если не задан, берётся title
  Высокая температура у взрослых

seoDescription: >                     # опционально — если не задан, берётся shortAnswer
  Что делать при температуре выше 38.5°C

faq:                                  # опционально
  - q: Какая температура считается высокой?
    a: Выше 38.5°C у взрослых.
```

---

## 7 блоков страницы

| # | Поле | Назначение |
|---|---|---|
| 1 | `shortAnswer` | Краткий ответ — что это, почему бывает. 1-2 предложения. |
| 2 | `keyPoints` | Главное — 3-5 ключевых фактов для быстрого понимания |
| 3 | `causes` | Возможные причины — список, от частых к редким |
| 4 | `selfCare` | Что можно сделать дома — конкретные действия |
| 5 | `whenToSeeDoctor` | Несрочные показания к врачу |
| 6 | `whenUrgent` | Красные флаги — когда нужна срочная помощь |
| 7 | `faq` | Часто задаваемые вопросы |

## Правила

- **slug совпадает с именем файла**: `high-fever.mdx` → `slug: high-fever`
- **category совпадает с папкой**: файл в `temperature/` → `category: temperature`
- **category должна существовать** в `src/content/categories/`
- **manualRelated** ссылается только на реально существующие slug'и
- **slug уникален** по всему сайту — не может быть двух `high-fever` в разных категориях
- **updated** обновляется при каждом редактировании
- **severity: high** автоматически добавляет красный блок «Срочная помощь» на страницу

Ошибки в этих правилах приводят к падению сборки с понятным сообщением — специально.

---

## Тоновые принципы

- Не ставим диагноз. Пишем «возможные причины», «может быть связано с».
- Не пугаем. Начинаем с частого и спокойного, заканчиваем редким и серьёзным.
- Не водим за нос. `shortAnswer` должен реально отвечать на вопрос заголовка за 30 секунд.
- Конкретно. «Пить больше жидкости» лучше, чем «соблюдать питьевой режим».

---

## Категории

Список актуальных категорий в `src/content/categories/`.
Добавление новой категории = новый `.mdx`-файл там же.

## Обязательная проверка перед созданием новой темы (Content Registry)

С этапа "Аудит существующего контента и Content Registry" в проекте есть единый машиночитаемый реестр всего контента: `src/data/content-registry.ts` (+ `content-registry.ids.json` и `content-registry.overrides.json`). Он не хранит вторую копию контента — title/slug/category/SEO всегда читаются напрямую из `.mdx`-файлов, реестр добавляет только id/status/quality/duplicateOf/related.

**Перед тем как писать новую страницу симптома — обязательно:**

```bash
npm run content:find -- "Название темы"
```

Это покажет, существует ли уже такая тема (EXACT MATCH), похожая тема (SIMILAR CONTENT) или ничего похожего нет. Полная документация — `docs/content-audit/CONTENT_AUDIT.md` и `docs/content-audit/overlaps.md` (там же — уже найденные дубли и смысловые пересечения, включая `sweating`/`excessive-sweating` и `blurred-vision`/`vision-loss`).

**После создания страницы и добавления её ID в `content-registry.ids.json`:**

```bash
npm run content:check
```

Команда завершится с ненулевым exit-кодом при дублях ID/URL/slug/title, битых ссылках и отсутствующих обязательных полях — публиковать/мёржить с непройденной проверкой нельзя.
